# Redis Sentinel Admin — Claude Code Guide

## Project Overview

A production-grade Golang **background service** for monitoring, diagnosing, and operating Redis Sentinel clusters on Kubernetes. Runs as a Pod inside the cluster, exposes a **REST API** for all data and operations. Designed to surface blind spots that generic monitoring tools (Prometheus, Grafana) miss — hot keys, config drift, replication lag trends, and pod-level connection attribution.

**Deployment model**: Deploy as a K8s Deployment alongside the Redis Sentinel cluster. Access via `kubectl port-forward` locally or expose via internal Ingress for team-wide access.

```
K8s Cluster
┌──────────────────────────────────────┐
│  redis-sentinel-admin (Pod)          │
│  ├── Core engine (polling + pub/sub) │
│  ├── K8s informer cache              │
│  └── REST API :8080                  │──── kubectl port-forward ──── curl / Postman / Swagger UI
└──────────────────────────────────────┘
         │ connects to
┌────────▼─────────────────────────────┐
│  Redis Sentinel Cluster              │
│  sentinel1:26379, sentinel2:26379    │
│  master:6379, replica1:6379, ...     │
└──────────────────────────────────────┘
```

## Core Conventions

### Go Code Style
- All exported types must have godoc comments
- Use `context.Context` as first parameter for all functions that touch Redis or K8s
- Errors must be wrapped with `fmt.Errorf("...: %w", err)` — never swallow errors
- No `panic()` outside of `main()` startup validation
- Use table-driven tests; minimum coverage for new packages: 70%

### REST API Conventions
- All responses in JSON, always wrapped: `{ "data": ..., "error": null }`
- Error response: `{ "data": null, "error": { "code": "NODE_UNREACHABLE", "message": "..." } }`
- Routes versioned: `/api/v1/...`
- Long-running data (key scanner, live events) exposed via **SSE** (`text/event-stream`)
- No mutation endpoints without explicit `"confirm": true` in request body
- **Every handler must have swaggo annotations** — see `skills/api-patterns.md`
- See `skills/api-patterns.md` for full conventions

### Redis Safety Rules
- **NEVER** use `KEYS *` in any code path — always use `SCAN` with cursor
- **NEVER** call `MONITOR` in production mode — use `SLOWLOG` or `OBJECT FREQ` instead
- All Redis calls must have explicit timeout via context: `ctx, cancel := context.WithTimeout(ctx, 2*time.Second)`
- Read operations on replicas must use `ReplicaOnly: true` client, never send reads to master unless explicitly intended

### Sentinel Connection Pattern
```go
// Always use this pattern — never connect directly to Redis nodes
rdb := redis.NewFailoverClient(&redis.FailoverOptions{
    MasterName:    cfg.MasterName,
    SentinelAddrs: cfg.SentinelAddrs,
    ReadTimeout:   2 * time.Second,
    WriteTimeout:  2 * time.Second,
})
```

### K8s Informer Pattern
```go
// Use informer cache — NEVER poll K8s API directly in hot paths
// See skills/k8s-integration.md for full setup
factory := informers.NewSharedInformerFactory(client, 30*time.Second)
podInformer := factory.Core().V1().Pods().Informer()
```

### Config Structure
All config via `config.yaml` + env override. Env prefix: `RSA_`.
Example: `RSA_SENTINEL_ADDRS=sentinel1:26379,sentinel2:26379`

### Logging
```go
// Use structured fields — no fmt.Printf in production code paths
logger.Info("replica lag detected",
    zap.String("replica", addr),
    zap.Int64("lag_bytes", lag),
    zap.Duration("elapsed", elapsed),
)
```

## Swaggo Workflow

```bash
# Install swag CLI (one-time)
go install github.com/swaggo/swag/cmd/swag@latest

# Regenerate docs after ANY change to handler annotations or main() @title block
swag init -g cmd/rsa-server/main.go -o docs/swagger

# Swagger UI available at:
# http://localhost:8080/swagger/index.html
```

**Rules**:
- Run `swag init` before committing whenever handler annotations changed
- Never manually edit `docs/swagger/` — it is fully generated
- All request/response structs used in annotations must be in `internal/api/` or `internal/api/dto/`

## Testing Conventions

- Unit tests: `_test.go` alongside source
- Integration tests: `integration/` directory, guarded by `//go:build integration`
- Use `miniredis` for Redis unit tests — no real Redis dependency in unit tests
- Mock K8s: use `k8s.io/client-go/kubernetes/fake`
- HTTP handler tests: use `net/http/httptest` + `echo.New()`

## Running Locally

```bash
# Prerequisites: Redis Sentinel running via docker-compose
cp config.example.yaml config.yaml

# Generate swagger docs
swag init -g cmd/rsa-server/main.go -o docs/swagger

# Start the service
go run ./cmd/rsa-server --config config.yaml
# Server starts on :8080

# Swagger UI
open http://localhost:8080/swagger/index.html

# Quick checks via curl
curl http://localhost:8080/api/v1/topology
curl http://localhost:8080/api/v1/replication/lag
curl http://localhost:8080/api/v1/diagnostics/slowlog
curl http://localhost:8080/api/v1/keys/hotkeys?top=20
curl http://localhost:8080/api/v1/keys/bigkeys?threshold_bytes=1048576

# SSE stream for live Sentinel events
curl -N http://localhost:8080/api/v1/events/stream
```

## Accessing in K8s

```bash
kubectl port-forward svc/redis-sentinel-admin 8080:8080 -n redis
open http://localhost:8080/swagger/index.html
```

| `RSA_SENTINEL_ADDRS` | `localhost:26379` | Comma-separated sentinel addresses |

| `RSA_METRICS_PORT` | `9090` | Prometheus metrics port |
| `RSA_POLL_INTERVAL` | `5s` | Polling interval for topology/replication |
| `RSA_SLOWLOG_INTERVAL` | `10s` | Slowlog pull interval per node |
